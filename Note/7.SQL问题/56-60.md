## 56. 可以基于一个视图再创建一个视图吗？

是的，可以基于一个视图创建另一个视图，这称为**嵌套视图**或**视图的视图**。 

**示例:**

```sql
-- 创建视图 View1
CREATE VIEW View1 AS
SELECT column1, column2
FROM table1
WHERE condition;

-- 基于 View1 创建视图 View2
CREATE VIEW View2 AS
SELECT *
FROM View1
WHERE other_condition;
```

**注意:** 

虽然可以创建嵌套视图，但过度使用嵌套视图可能会导致性能问题，并增加代码的复杂性，使其难以理解和维护。 建议尽量避免创建过多的嵌套视图层级。


## 57. 原表删除后还可以使用视图吗？

不可以。视图是从一个或多个基表派生出来的虚拟表。 如果删除了基表，视图将失去数据源，因此将无法使用。 

当你尝试查询或访问一个基于已删除基表的视图时，数据库系统通常会返回一个错误消息，指示该视图所依赖的对象不存在。


## 58. 你知道哪些类型的 SQL 关系？

在关系型数据库中，表之间可以通过**键**来建立关系，以反映现实世界中实体之间的联系。常见的 SQL 关系类型包括：

* **一对一关系 (One-to-One)**: 一个表中的每一行记录只与另一个表中的一行记录相关联。 例如，一个员工可以有一张员工卡，一张员工卡也只能属于一个员工。
* **一对多关系 (One-to-Many)**: 一个表中的一行记录可以与另一个表中的多行记录相关联，反之则不行。 例如，一个客户可以有多个订单，但一个订单只能属于一个客户。
* **多对多关系 (Many-to-Many)**: 一个表中的多行记录可以与另一个表中的多行记录相关联。 例如，一个学生可以选择多门课程，一门课程也可以被多个学生选择。


## 59. BOOLEAN 数据字段的值有哪些？

BOOLEAN 数据类型表示逻辑真值，通常用于存储 `TRUE` 或 `FALSE` 值。 

* 在 **PostgreSQL** 中，BOOLEAN 类型可以接受的值包括：
    * `TRUE`
    * `FALSE`
    * `NULL` (表示未知)
* 在其他一些数据库系统中，例如 **MySQL** 和 **SQLite**，没有专门的 BOOLEAN 类型，通常使用 `TINYINT(1)` 来表示布尔值，其中：
    * `1` 表示 `TRUE`
    * `0` 表示 `FALSE`
* 在 **SQL Server** 中，使用 `BIT` 数据类型来存储布尔值，其中：
    * `1` 表示 `TRUE`
    * `0` 表示 `FALSE`


## 60. SQL 中的范式 (Normalization) 是什么，为什么要使用范式？

数据库范式 (Database Normalization) 是一组用于设计关系型数据库的规则和指南，旨在减少数据冗余、提高数据一致性和完整性。 

**为什么要使用范式?**

* **减少数据冗余**:  范式化可以消除数据库中重复的数据，节省存储空间并提高查询效率。
* **提高数据一致性**:  范式化可以避免数据更新异常，确保数据的一致性和准确性。
* **提高数据完整性**:  范式化可以确保数据之间的关系正确，防止数据丢失或不一致。
* **提高查询效率**:  范式化可以简化查询语句，提高查询性能。
* **提高数据灵活性**:  范式化可以使数据库更易于扩展和修改，适应不断变化的需求。

**常见的数据库范式:**

* **第一范式 (1NF)**: 要求数据库表中的每一列都必须是原子的，不可再分的数据单元。
* **第二范式 (2NF)**: 在满足第一范式的基础上，要求所有非主键列完全依赖于主键，即不存在部分依赖。
* **第三范式 (3NF)**: 在满足第二范式的基础上，要求非主键列之间没有传递依赖。

**注意:** 

数据库范式化是一个权衡的过程，需要根据实际情况进行选择。 过度范式化可能会导致数据库结构过于复杂，降低查询性能。 在实际应用中，通常会根据数据库的规模、性能要求和数据一致性要求来选择合适的范式级别。
