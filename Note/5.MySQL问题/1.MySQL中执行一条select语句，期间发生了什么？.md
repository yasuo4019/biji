MySQL 执行一条 `SELECT` 查询语句时，会经历多个步骤，这些步骤可以分为两层：Server 层和存储引擎层。以下是详细的流程和各个功能模块的作用：

### 1. Server 层与存储引擎层
- **Server 层**：负责建立连接、解析、优化和执行 SQL 语句，包含了连接器、查询缓存、解析器、优化器和执行器等核心模块。
- **存储引擎层**：负责数据的存储和提取，常见的存储引擎有 InnoDB 和 MyISAM。InnoDB 是 MySQL 的默认存储引擎。

### 2. 查询语句的执行流程

#### 2.1 连接器
- **TCP 连接**：客户端与 MySQL 通过 TCP 三次握手建立连接。MySQL 验证用户名和密码，成功后读取用户权限，随后用户可以执行 SQL 语句。
- **连接管理**：`show processlist` 命令可以查看当前 MySQL 服务被多少客户端连接。MySQL 支持的最大连接数由 `max_connections` 参数控制，超过该值系统会拒绝新的连接请求。

#### 2.2 查询缓存
- **缓存查找**：MySQL 在接收到 `SELECT` 查询语句后，会首先检查查询缓存（如果存在）。如果缓存中有相应的结果，直接返回结果；如果没有，则继续执行查询。
- **MySQL 8.0 变化**：从 MySQL 8.0 开始，查询缓存被移除，查询不会再走缓存步骤。

#### 2.3 解析器
- **词法分析**：解析器将 SQL 语句分解成关键字、表名、字段名等基本元素。
- **语法分析**：根据词法分析的结果，检查 SQL 语句是否符合语法规则，并构建 SQL 语法树。

#### 2.4 预处理器
- **预处理**：检查 SQL 语句中的表和字段是否存在，并扩展 `SELECT *` 为具体的列名。如果表或字段不存在，会在此阶段报错。

#### 2.5 优化器
- **执行计划**：优化器根据查询语句的复杂度、表的索引情况，制定最优的执行计划，选择使用哪个索引（如主键索引、二级索引等）。

#### 2.6 执行器
- **执行 SQL**：执行器按照优化器制定的执行计划，逐步与存储引擎交互，执行查询并将结果返回给客户端。执行过程涉及多种操作，如主键索引查询、全表扫描等。

### 3. 执行过程中的细节

#### 3.1 主键索引查询
- **通过主键索引快速定位记录**：执行器调用存储引擎的接口，通过主键索引定位到记录，并返回给客户端。

#### 3.2 全表扫描
- **逐行扫描**：如果查询语句不涉及索引，存储引擎会逐行扫描表中的记录，执行器判断记录是否符合查询条件，符合条件的记录将被发送给客户端。

#### 3.3 索引下推
- **优化查询效率**：索引下推是一种优化策略，将部分过滤操作推送到存储引擎层，减少不必要的回表操作，提高查询效率。

### 总结
MySQL 的查询执行过程展示了其内部复杂的架构设计。通过了解 Server 层和存储引擎层的分工以及每个功能模块的作用，能更好地理解 MySQL 的执行机制，并在实际应用中优化查询性能。
